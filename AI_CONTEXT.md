# Контекст Проекта: HackNU AI (2026)

Этот файл предназначен для ввода любого AI-ассистента в курс дела. Прочитайте его перед тем, как вносить изменения в архитектуру или логику приложения.

## 🎯 Суть проекта
**HackNU AI** — это платформа для совместной работы (коллаборативный workspace), объединяющая в себе интерактивный визуальный холст (node-граф) и продвинутые AI-инструменты. Предназначена для командного брейншторминга, генерации контента и архитектурного проектирования.
Проект создается для хакатона (HackNU 2026).

## 📊 Основной функционал
1. **Интерактивный Холст (Canvas):** Рабочая область на базе нодов (построенная на `React Flow`), где пользователи создают графы. Включает специализированные узлы (например, `HiggsfieldNode`), поддерживающие генерацию медиа (текст-в-видео).
2. **AI-интеграция:** 
   - **Gemini Chat:** Встроенный ассистент, который "видит" состояние доски и помогает команде.
   - **Higgsfield (AI Media):** Генерация изображений и видео по промптам прямо на холсте.
3. **Голосовая и видеосвязь в реальном времени:** Интеграция с **LiveKit** для пространственного аудио и видео.
4. **Синхронизация (Real-time Collaboration):** Совместная работа над доской (а-ля Figma) через WebSockets (Laravel Reverb).

## 🏗 Наш Стек технологий
### Frontend
- **Ядро:** React 18, Vite.
- **Роутинг:** `react-router-dom` v6.
- **Архитектура:** **FSD (Feature-Sliced Design)**. Строгое разделение слоев:
  - `app/` — Инициализация приложения, глобальные стили, роутеры (`App.jsx`).
  - `pages/` — Страницы приложения (`AuthPage`, `ProjectHubPage`, `WorkspacePage`). **Важно:** страничный роутинг использует прямые импорты из папки `ui/` компонентов.
  - `widgets/` — Самостоятельные крупные блоки (например, `<Canva />`).
  - `features/` — Бизнес-логика (взаимодействие с пользователем).
  - `entities/` — Сущности предметной области (пользователи, проект, сессии).
  - `shared/` — Общие UI-компоненты, API инстансы, утилиты.
- **Отрисовка графов:** `@xyflow/react` (React Flow).
- **Стилизация:** Tailwind CSS. 
  - *Дизайн-код:* Киберпанк / Futuristic Dark Mode. Глубокие фоны (`neutral-900`, `neutral-950`), акценты циана (`cyan-400`), фуксии (`fuchsia-500`) и лайма (`lime-500`). Использование `backdrop-blur`.

### Backend (Reference)
- Семейство **Laravel**.
- **БД:** PostgreSQL.
- **Аутентификация:** Laravel Sanctum.
- **WebSockets:** Laravel Reverb (для реалтайм курсоров и обновлений `Canvas`).

## 🚨 Строгие Правила Разработки для AI
1. **FSD Архитектура:** Никогда не нарушай правило зависимостей FSD (нижние слои не могут импортировать верхние). 
2. **Экспорты страниц:** Используй только **именованные экспорты** (`export const WorkspacePage = ...`) для компонентов внутри FSD, чтобы избежать проблем Vite HMR с кешированием на macOS.
3. **Стиль кода (UI):** Сохраняй текущий дизайн. Если добавляешь новую кнопку или панель, она должна использовать классы Tailwind из нашей палитры (стекло + темный неон). Никаких белых дефолтных блоков в воркспейсе.
4. **Импорты:** Будь внимателен к регистру файлов на macOS (например, не создавай `WorkSpace.jsx`, если папка называется `workspace` или если уже есть `WorkspacePage.jsx`).
5. **LLM/API:** Все ключи для AI (OpenRouter, Higgsfield) должны вызываться через защищенные эндпоинты (Backend), или, по необходимости демо-версии, подгружаться из `.env`.

## 🔄 User & Data Flow (Маршрутизация и Состояния)

Чтобы понимать, как данные текут от логина до холста, ознакомьтесь со следующей логикой:

### 1. State Management (Управление состоянием)
- **Токен сессии:** Хранится в `localStorage` (доступ через `shared/lib/storage.js`). В `App.jsx` инициализируется состояние `sessionConfig`, которое передается пропсами вниз по дереву: `<ProjectHubPage sessionConfig={sessionConfig} />` и `<WorkspacePage sessionConfig={sessionConfig} />`.
- **Глобальные стейт-менеджеры (Zustand/Redux):** Пока *не используются* для глобальной передачи данных. Все идентификаторы сущностей (например, `roomId`) извлекаются из **URL параметров** (через `react-router-dom` `useParams()`), а токен — из пропсов `sessionConfig`.
- **Локальные состояния холста (Controlled Component):** Стейт графа (`nodes`, `edges`) поднят на уровень страницы `WorkspacePage`. Граф React Flow внутри `<Canva />` теперь полностью контролируемый и принимает `nodes` и `edges` через пропсы. Это позволяет странице оркестрировать ИИ-генерацию карточек и пушить их напрямую на холст.

### 2. Lifecycle зависимостей (Жизненный цикл комнаты)
При заходе на роут `/room/:id` (через клик по карточке в `ProjectHubPage`):
1. **Передача ID:** Компонент `<WorkspacePage />` рендерится. Идентификатор комнаты `id` доступен из параметров URL (`/room/1` -> `id: 1`).
2. **Токен сессии:** `sessionConfig` вливается из `App.jsx`, гарантируя, что мы имеем аутентификацию (если токена нет, происходит автоматический редирект `Navigate to="/"`).
3. **API & Инициализация (Hydration):** 
   - Выполняется `GET /api/rooms/{id}`. Безопасно парсится вложенный `canvas_state` (с учетом оберток Axios `payload.data`). Идет гидратация локального стейта доски готовыми карточками.
   - Включается `useEffect` с Debounce-таймером (1.5 сек), который автоматически делает POST-сохранение всех мутаций холста в БД.
   - Ожидается: вызов `fetchVideoToken` для подключения LiveKit (плавающий виджет).
   - Ожидается: примонтировать WebSockets: `Echo.join('room.${roomId}')` для реалтайм курсоров коллег.

### 3. Связь API и UI (`shared/api/platform.js`)
В `platform.js` определено централизованное API для работы с сущностями. Основные цепочки:
- **Аутентификация -> Проекты:** `fetchCurrentUser` -> `fetchProjects` -> `fetchRooms`.
- **Генерация медиа AI (Higgsfield):** 
  Отправляется `generateRoomMedia` с промптом ➝ Бэкенд возвращает `requestId` ➝ Фронтенд начинает пуллить через `fetchGenerationStatus` до статуса `completed` ➝ Нода на холсте обновляется сгенерированным контентом.
- **AI Ассистент (Spatial Participant - Gemini):**
  Пользователь взаимодействует с плавающим виджетом `FloatingAICommand` (правый сайдбар удален в угоду Full-Screen погружению). Нажатие "отправить" форвардит текущий `canvas_state` и промпт в бэкенд контроллер.
  LLM возвращает строгий JSON, содержащий `new_nodes`. Карточки немедленно вклеиваются в `setNodes`, рендерятся и сейвятся дебаунсером в базу данных!
